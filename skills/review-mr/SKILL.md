# review-mr

深度分支代码审查，核心关注回归安全和架构一致性。

## 参数解析

解析用户在 `/review-mr` 后输入的参数：

| 输入 | 行为 |
|------|------|
| `<特性分支> <基准分支>` | Review 特性分支相对于基准分支的改动 |
| `<特性分支>` | Review 特性分支相对于默认基准分支（master 或 main）的改动 |
| （空） | Review 当前分支相对于默认基准分支（master 或 main）的改动 |

解析步骤：

1. 传入两个参数：第一个为特性分支，第二个为基准分支
2. 传入一个参数：作为特性分支；自动检测基准分支（`master` 或 `main`，取存在的那个）
3. 无参数：使用当前分支（`git branch --show-current`）；自动检测基准分支
4. 优先检查本地分支，不存在则尝试 `origin/<branch>` 远程引用
5. 需要时自动 fetch：`git fetch origin <branch>`
6. 分支不存在时给出明确错误提示并终止

## Review 流程

按以下阶段顺序执行，使用任务管理工具追踪进度。

### 阶段一：变更范围分析

```bash
# 提交历史（排除 merge commits）
git log --oneline --no-merges <base>..<feature>

# 文件变更概览
git diff --stat <base>..<feature>

# 完整 diff
git diff <base>..<feature>
```

输出变更摘要：
- 提交数（排除 merge）、变更文件数、增删行数
- 按模块/子包归类（将文件映射到架构组件）
- 区分新增文件与修改文件

### 阶段二：意图理解

在审查代码质量之前，先深入理解**改动要实现什么目标**：

1. 阅读所有 commit message（含正文），理解改动叙事
2. 完整阅读每个新增文件，理解引入的新模块/抽象
3. 阅读每个修改文件的完整 diff，并结合基准版本的上下文
4. 识别核心特性/修复的目标
5. 用 2-3 句话总结改动意图

### 阶段三：架构与设计审查

对照以下维度评估：

- **项目架构规则**：阅读项目的 AGENTS.md / CODEBUDDY.md 及子包 AGENTS.md，了解编码规范和禁止模式
- **模块边界**：改动是否尊重现有依赖方向？是否引入循环依赖？
- **关注点分离**：改动是否在正确的层级？是否混合了应分离的关注点？
- **设计模式**：是否遵循代码库已有的模式，还是引入了不必要的不一致？
- **API 设计**：新接口/类型命名是否清晰、最小化、与现有约定一致？
- **业界最佳实践**：实现方案是否为该技术栈的标准做法？

### 阶段四：实现质量审查

检查以下方面：

- **正确性**：逻辑错误、边界值错误、null/undefined 处理、竞态条件
- **错误处理**：所有错误路径是否覆盖？错误传播/吞没是否合理？
- **类型安全**：是否引入 `any`、`@ts-ignore`、`@ts-expect-error`？类型收窄是否正确？
- **资源管理**：内存泄漏、未关闭的 handle、定时器清理、事件监听器清理
- **边界场景**：空输入、边界值、并发访问、超时场景
- **重复实现**：是否重新实现了代码库中已有的功能？
- **死代码**：未使用的变量、不可达分支、注释掉的代码
- **一致性**：命名约定、代码风格、import 排序是否与项目一致

### 阶段五：回归与影响分析（最关键）

这是最高优先级的阶段。系统性检查：

1. **函数签名变更**：是否有现有调用方因参数变更而破坏？
2. **行为变更**：任何现有代码路径在改动后是否产生不同结果？
3. **导入变更**：是否移除了其他模块依赖的导出？
4. **共享状态**：改动是否修改了其他代码读取的全局/模块级状态？
5. **模块加载副作用**：是否有顶层代码执行影响其他模块？
6. **配置/常量变更**：变更的值是否影响其他功能？
7. **错误处理变更**：变更的错误处理是否可能导致之前被捕获的错误向上传播？
8. **时序变更**：新增的异步操作是否可能改变执行顺序？
9. **性能影响**：改动是否可能降低现有热路径的性能？

对每个变更文件明确回答："此改动是否可能影响任何现有功能？如果是，如何影响？"

当改动是纯粹的新增（新文件、新函数、包裹在 try/catch 中且不 throw），应如实说明，而非制造虚假风险。

### 阶段六：完整性检查

验证：

- **功能完整性**：实现是否覆盖了预期功能的所有方面？
- **漏改检查**：是否有应该修改但未修改的文件？（如类型定义、测试、配置）
- **部分实现**：是否遗留 TODO/FIXME？是否有桩函数？是否有应可配置的硬编码值？
- **文档准确性**：注释是否与代码一致？是否有误导性注释？

## 报告格式

输出以下结构化报告：

```markdown
# MR Review 报告: <特性分支> vs <基准分支>

## 1. 变更概览

| 指标 | 值 |
|------|-----|
| 提交数 | N（排除 merge） |
| 变更文件 | N（新增: X, 修改: Y） |
| 增删行数 | +N / -N |
| 涉及模块 | 列表 |

### 提交摘要
- commit-hash — message（每个 commit 一行）

### 改动意图
2-3 句话总结改动目标。

## 2. 架构与设计评估

[架构一致性、设计质量、模式一致性评估]

整体评价: [优秀 / 合格 / 需改进 / 拒绝]

## 3. 问题清单

### P0 — 严重（合入前必须修复）
会导致 bug、数据丢失、安全漏洞或破坏现有功能的问题。

### P1 — 高风险（强烈建议修复）
影响新功能正确性、重大代码质量问题或潜在回归风险。

### P2 — 中等（应当修复）
不一致、误导性注释、次优模式、次要边界场景。

### P3 — 低风险（可选 / 风格）
风格建议、命名建议、微小优化。

每个问题格式：

#### N.M <标题>
**文件**: `path/to/file.ts:LINE`
**描述**: 问题是什么，为什么重要。
**建议**: 如何修复。

## 4. 回归风险评估

| 影响面 | 风险等级 | 分析 |
|--------|---------|------|
| 区域 | 无/低/中/高 | 说明 |

## 5. 完整性检查

| 检查项 | 状态 | 备注 |
|--------|------|------|
| 功能完整实现 | 是/部分/否 | 详情 |
| 是否存在漏改 | 是/否 | 详情 |
| 注释准确性 | 是/否 | 详情 |

## 6. 合入决策

### 结论: [通过 / 有条件通过 / 需修改 / 拒绝]

**总结**: 1-2 句整体评价。

**合入前必须修复**（如有）:
1. 编号列表

**建议改进**（如有）:
1. 编号列表
```

## 审查原则

- 客观公正，不夸大也不淡化问题。
- 设计良好的部分简要认可，不堆砌无意义的赞美。
- 明确区分"阻塞合入"问题和"锦上添花"改进。
- 如果改动是纯旁路新增且有完善隔离（try/catch、fire-and-forget），在风险评估中如实反映，不制造虚假风险。
- 使用代码片段说明问题，引用具体文件路径和行号。
- 对严重程度不确定的问题，宁可标记让作者判断，不要遗漏。
- 功能实现且优雅是基础，绝对不能影响现有功能是底线——这是 review 的第一原则。
